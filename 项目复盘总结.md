# TTS Studio — 开发复盘

> 基于 MOSS-TTS-Nano 的 Web 语音合成平台，从 ONNX 模型集成到完整的前后端克隆+TTS 功能。

---

## 一、项目概况

- **目标**：搭建一个 Web 端语音合成平台，支持内置音色 TTS + 声音克隆
- **技术栈**：Python FastAPI + ONNX Runtime + React + Vite + Tailwind CSS
- **模型**：MOSS-TTS-Nano-100M（5 个 ONNX session）+ MOSS-Audio-Tokenizer
- **时间跨度**：从模型下载到全功能验证通过

---

## 二、错误清单与修复

### 1. 模型加载：torchaudio 的 torchcodec 后端不兼容

**现象**：`_load_reference_audio()` 调用 `torchaudio.load()` 返回空张量，导致克隆音频为静音。

**根因**：PyTorch 2.11 默认启用 `torchcodec` 后端，需要 FFmpeg DLL 解码 MP3。Windows 系统未安装 FFmpeg 时解码失败，返回全零张量。

**解决**：将 `torchaudio.load()` 替换为 `soundfile.read()`，后者不依赖 FFmpeg 且支持常见音频格式。

**文件**：`temp_moss_tts/onnx_tts_runtime.py` — `_load_reference_audio()`

```python
# 修复前：torchaudio.load() 依赖 FFmpeg DLL，缺失时无声返回全零
# torchaudio 2.11 默认启用 torchcodec 后端，Windows 无 FFmpeg 则解码失败
audio_data, sample_rate = torchaudio.load(str(path))

# 修复后：改用 soundfile.read()，纯 Python 实现，不依赖外部解码器
# soundfile 通过 libsndfile 直接解析 WAV/MP3/FLAC 等常见格式
import soundfile as sf
audio_data, sample_rate = sf.read(str(path))
```

**教训**：第三方库的默认行为可能随版本变化。`torchaudio` 的 backend 切换是静默的，没有报错，只返回空数据。在音频处理中优先使用底层库（`soundfile`、`wave`）而非高层封装。

### 2. 无声输出：target_format 取了用户格式写入 `.wav` 路径

**现象**：选择 MP3 格式合成时，前端预览无声；下载的 `.wav` 文件也无法播放。

**根因**：`_synthesize_official()` 从 `voice_config` 中读取 `format` 字段（用户选择的格式，如 `"mp3"`），然后以此格式写入到 `output_path`（路径以 `.wav` 结尾）。结果 `.wav` 文件里装的是 MP3 数据。浏览器通过扩展名判断格式为 WAV，用 WAV 解码器播放 MP3 数据，结果无声。

**解决**：中间文件始终输出 WAV 格式，最终格式转换由 `_run_synthesis` 统一处理。

**文件**：`backend/app/models/moss_tts.py` — `_synthesize_official()`

```python
# 修复前：从 voice_config 读取用户选择的格式（如 "mp3"）直接写入 output_path
# output_path 以 .wav 结尾，但写入 MP3 数据 → 浏览器解码为 WAV 失败 → 无声
target_format = voice_config.get("format", "wav")

# 修复后：中间文件始终输出 WAV，最终的 MP3/PCM 格式转换由 _run_synthesis 统一处理
# 分离"模型生成"和"格式转换"两个步骤，避免格式污染
target_format = "wav"
```

**教训**：文件扩展名必须与内容格式一致。pydub 的 `from_file()` 会自动探测格式（不看扩展名），所以 API 测试时用 pydub 验证"通过"了，但浏览器 `<audio>` 依赖扩展名。测试工具的选择会掩盖生产环境的 bug。之后在测试中增加了文件头部字节校验（RIFF / ID3）。

### 3. 无声输出：后端子进程缺少 `cwd` 参数

**现象**：通过 API 调用内置音色合成时，长文本（>50 字）返回无声文件；直接命令行运行 `infer_onnx.py` 却正常。

**根因**：`_run_inference()` 调用 `subprocess.run()` 时没有指定 `cwd`，子进程在当前工作目录（backend/）启动。`infer_onnx.py` 以当前目录为基准查找部分资源，目录不匹配时模型加载失败，子进程返回错误码。`_run_inference` 检测到失败后返回 `None`，触发 `_synthesize_stub()` 生成静音文件。

而 `_run_inference_clone()`（克隆音色路径）一直有 `cwd=str(temp_moss_dir)`，所以克隆合成正常。

**解决**：为 `subprocess.run()` 添加 `cwd=str(temp_moss_dir)`。

**文件**：`backend/app/models/moss_tts.py` — `_run_inference()`

```python
# 修复前：未设置 cwd，子进程在当前工作目录（backend/）启动
# infer_onnx.py 依赖 temp_moss_tts/ 目录下的资源文件，目录不匹配导致模型加载失败
result = subprocess.run(cmd, capture_output=True, text=True, env=env, timeout=180)

# 修复后：指定 cwd 为 temp_moss_tts 目录，与 _run_inference_clone() 保持一致
# 子进程在正确的目录中启动，能正常找到模型配置文件
result = subprocess.run(cmd, capture_output=True, text=True, env=env,
                        cwd=str(temp_moss_dir), timeout=180)
```

**教训**：子进程的 `cwd` 容易被忽略。新旧两个方法（`_run_inference` 和 `_run_inference_clone`）写于不同时间，一个加了 `cwd` 一个没加，导致了不一致的行为。对比两个方法的参数差异能快速定位问题。

### 4. 合成超时：子进程 180s 不够

**现象**：295 字长文本合成时，子进程被 `TimeoutExpired` 杀死，任务失败。

**根因**：ONNX 模型在 CPU 上每帧生成约 0.4-0.5 秒，722 帧需要约 340 秒。180s 和 300s 的超时均不足以完成长文本合成。

**解决**：将内置音色子进程超时从 180s 提升到 600s；`--max-new-frames` 从 200 提升到 900。

**文件**：`backend/app/models/moss_tts.py` — `_run_inference()`

**教训**：CPU 上的 ONNX 推理速度是瓶颈。长文本的帧数可以估算（~2.5 帧/字 × 字数），超时应留有足够余量。实测 295 字合成需约 6 分钟，应提前告知用户。

---

### 5. 男声最后一句缺失：chunk 过大导致 EOS 过早触发

**现象**：小刚（男声）合成 295 字文本时，只念到"至此"就结束了，最后一句"ONE完成了……彻底转身"丢失。小媛（女声）合成同一文本完整。

**根因**：`infer_onnx.py` 默认 `--voice-clone-max-text-tokens 75`，295 字被拆成 4 个 chunk。分句边界不均衡导致最后一个 chunk 约 120 字。模型在处理超大 chunk 时，生成到一半提前输出了 EOS token，剩余文本被跳过。女声可能因为语速或声学特性不同，更不容易触发此问题。

**解决**：内置音色子进程添加 `--voice-clone-max-text-tokens 40`，缩小 chunk 大小，降低单 chunk 内提前 EOS 的概率。

**文件**：`backend/app/models/moss_tts.py` — `_run_inference()`

```python
# 添加 --voice-clone-max-text-tokens 参数控制 chunk 大小
# 默认 75 tokens → 分 4 个 chunk，最后一个 chunk 约 120 字 → 模型过早 EOS
# 改为 40 tokens → 拆成 7-8 个更小的 chunk，每个 chunk 在模型能力范围内完整合成
"--voice-clone-max-text-tokens", "40",
```

**教训**：不同音色的模型行为可能存在差异。男声和女声的声学模型参数不同，对 chunk 大小的容忍度也不同。不能因为一个音色通过测试就认为所有音色都正常。

### 6. 音色列表重复：`clonedVoices` prop 与 API 返回未去重

**现象**：切换到"声音克隆"再切回 TTS 页面，音色列表中的选项变多。每次切换模式都增加。

**根因**：`VoiceSelector` 挂载时通过 `useEffect` 从 `/api/voices` 拉取音色列表，该 API 也返回了克隆音色（通过 `voice_clone_service.list_cloned_voices()`）。同时 `App.tsx` 通过 `clonedVoices` prop 又传入了相同的克隆音色。两者合并为 `allVoices` 时没有按 `id` 去重，导致克隆音色重复出现。

**解决**：合并数组时按 `id` 去重。

**文件**：`frontend/src/components/VoiceSelector.tsx`

```typescript
// 修复前：直接合并 API 返回的音色列表和本地克隆音色列表
// API 已包含克隆音色（voice_clone_service.list_cloned_voices()），
// clonedVoices prop 也包含相同数据 → 重复显示
const allVoices = [...voices, ...clonedVoices];

// 修复后：按 id 去重，保留首次出现的条目（API 返回的优先，clonedVoices 补充）
// arr.findIndex(x => x.id === v.id) === i 确保每个 id 只保留第一个匹配项
const allVoices = [...voices, ...clonedVoices].filter(
  (v, i, arr) => arr.findIndex(x => x.id === v.id) === i
);
```

**教训**：前端数据来自多个源头时，必须去重。常见的模式：API 返回 + 本地状态。如果后端 API 也返回了本地状态中已存在的数据，合并时必须去重。

### 7. 音频播放无声：浏览器自动播放策略

**现象**：合成完成后，在 AudioPlayer 中点击播放按钮，无声音输出。控制台无报错。

**根因**：`AudioPlayer` 中 `audioRef.current.play()` 返回的 Promise 被浏览器拒绝（自动播放策略），但拒绝未被捕获，用户看不到错误提示。

**解决**：为 `play()` 添加 `.catch()` 处理，显示错误提示文字。

**文件**：`frontend/src/components/AudioPlayer.tsx`

```typescript
// 浏览器自动播放策略会拒绝 audio.play() 返回的 Promise
// 不加 .catch() 的话，拒绝静默发生，用户看不到任何提示
// 捕获后显示中文引导文字，提示用户手动点击播放
audioRef.current.play().catch(() => {
  setAudioError('浏览器已阻止自动播放，请点击播放按钮');
});
```

**教训**：浏览器自动播放策略是所有 Web 音频应用都会遇到的问题。`HTMLAudioElement.play()` 返回 Promise，必须处理拒绝情况。解决方案：在用户手势回调中直接调用 `play()`（而非通过状态变更间接调用），或显示明确的错误引导。

### 8. 测试工具掩盖问题：pydub 自动探测格式

**现象**：API 测试用 `pydub.AudioSegment.from_file()` 验证音频，始终报告 RMS > 1000（"有声音"）。但用户在前端听到的是无声。

**根因**：pydub 依赖 ffmpeg 自动探测文件格式，不看扩展名。即使 `.wav` 文件里装的是 MP3 数据，pydub 也能正确解码并报告 RMS 正常。但浏览器 `<audio>` 依赖扩展名和 Content-Type 来选择解码器，对于 `.wav` 扩展名但 MP3 内容的文件，WAV 解码器无法解码，输出无声。

**解决**：在测试中增加文件头部字节校验：

```python
# 读取文件前 4 字节验证 WAV 头部是否为 RIFF
# pydub 用 ffmpeg 自动探测格式会掩盖"WAV 文件装了 MP3 数据"的问题
# 直接检查文件二进制头部是最可靠的校验方式——不依赖任何解码器
with open(f, 'rb') as fp:
    header = fp.read(4)
    # RIFF = 有效 WAV，b'ID3' / b'\xff\xfb' = MP3 数据被错误写入 .wav 文件
    assert header == b'RIFF', f"Not a valid WAV: {header}"
```

**文件**：`backend/tests/test_api.py` — `test_task_status_flow_wav`、`test_task_status_flow_mp3`

**教训**：测试工具的行为可能与被测系统的行为不同。pydub/ffmpeg 非常宽容，但浏览器严格遵循文件规范。测试应该尽量模拟真实使用环境，或用文件头部校验作为补充断言。

### 9. 长文本合成变慢：ONNX CPU 推理速度

**现象**：短文本（<50 字）合成约 15-30 秒完成。长文本（295 字）合成需要约 5-6 分钟。

**根因**：MOSS-TTS-Nano 的 ONNX 模型在没有 GPU 的情况下，每帧推理约需 0.4-0.5 秒。295 字约需 722 帧，因此总时间约 340 秒。这是 ONNX Runtime 在 CPU 上的固有限速。

**影响**：用户需要等待较长时间才能获得长文本的合成结果。子进程超时需要设置为足够大（600s），前端轮询也需要相应延长。

**教训**：对 CPU 推理应有合理的速度预期。可在 UI 中显示预估等待时间，或提供一个进度提示（当前帧/总帧数）。这是模型的固有限制，无法通过代码优化解决。

---

### 10. 声音克隆预览生成取消后残留

**现象**：（重构前）克隆成功后，`CloneVoicePanel` 自动调用 `generatePreviewAudio()` 生成预览音频。当 `handleCloneSuccess` 切换到 TTS 模式后，`CloneVoicePanel` 被卸载，但预览生成的异步任务仍在运行，对已卸载的组件调用 `setState`。

**根因**：`handleClone` 中 `onSuccess()` 触发模式切换（`setMode('tts')`），导致 `CloneVoicePanel` 卸载。但之后的 `setClonedResult()` 和 `generatePreviewAudio()` 仍然执行，在已卸载的组件上调用 `setState`。

**解决**：移除预览音频生成逻辑。克隆成功后直接通过 `onSuccess` 注册音色并切换到 TTS 模式，由用户在 TTS 页面自行合成试听。

**文件**：`frontend/src/components/CloneVoicePanel.tsx`

**教训**：异步操作的生命周期需要与组件生命周期同步。在组件卸载后继续调用 `setState` 在 React 18 中虽然不会崩溃，但会在开发控制台产生警告，且可能导致内存泄漏。

---

### 11. 系统死机：ONNX 多进程内存耗尽

**现象**：同时提交多个合成请求后，系统变得极度缓慢，鼠标移动卡顿，任务管理器显示内存占用 >95%，最终系统无响应，只能强制重启。

**根因**：每次子进程调用都会加载一份完整的 ONNX 模型（约 4.5GB）。后端 FastAPI 本身也加载了一份模型（约 4.5GB）。两进程加起来约 9GB。加上系统（Windows 约 3-4GB）和浏览器/IDE 等进程，16GB 内存被完全耗尽。当内存不足时，系统开始使用页面文件（硬盘交换），导致所有进程变慢，最终死机。

`--cpu-threads 4` 加剧了问题：每个子进程创建 4 个线程，多进程并发时线程数激增，内存和 CPU 上下文切换开销进一步放大。

**解决**：

1. `TTSService` 设置 `max_workers=1`（`ThreadPoolExecutor` 单线程），确保同一时间只有一个合成任务在执行。
2. 子进程参数从 `--cpu-threads 4` 改为 `--cpu-threads 2`，减少每进程的内存和线程开销。
3. 在 `_run_synthesis()` 中添加预检内存检查：

```python
# 合成前检查可用内存：ONNX 模型需要约 4.5GB，后备至少保留 2GB
# 低于阈值时拒绝请求，避免因内存不足导致系统死机
# psutil.virtual_memory() 返回系统级内存状态，比任务管理器更精确
avail_gb = psutil.virtual_memory().available / (1024 ** 3)
if avail_gb < 2.0:
    raise MemoryError(f"Insufficient memory: only {avail_gb:.1f}GB available (need 2GB+)")
```

4. 输入音频长度限制：上传时检查音频时长，超过 30 秒的音频提示用户裁剪。

**文件**：`backend/app/services/tts_service.py`、`backend/app/models/moss_tts.py`

**教训**：

- ONNX Runtime 的模型加载不是轻量操作。每个进程/线程都会加载独立的一整套 session，内存占用线性增长。
- `ThreadPoolExecutor` 的默认 `max_workers` 是 CPU 核心数 × 5，这对 I/O 密集型任务合理，但对内存密集型任务极其危险。必须显式限制并发数。
- 子进程方案虽然隔离了模型加载对后端进程的影响，但付出了双倍内存代价。如果系统内存不足，应考虑单进程方案。
- 压力测试不能跳过。单请求正常不代表多请求正常。内存泄漏和并发问题只在负载下才会暴露。

**改进方向**：考虑使用共享内存或 ONNX Runtime 的 session 复用机制，避免多进程各自加载模型。或者将子进程改为守护进程模式（常驻模型），通过 stdin/stdout 通信，消除每次请求的模型加载开销。

## 三、排查方法总结

### 3.1 问题优先级

| 优先级 | 判断标准 | 做法 |
|--------|---------|------|
| P0 | 用户明确报告、可稳定复现 | 立即排查，模拟用户操作复现 |
| P1 | 测试发现、影响核心流程 | 定位根因后修复，补充测试 |
| P2 | 偶发、边缘情况 | 记录日志，增加检测，批量修复 |

### 3.2 排查步骤（经验沉淀）

1. **还原现场**：复现用户的操作序列，确认问题是否存在。
2. **缩小范围**：用 API 直接调用（绕过前端）测试后端，用命令行（绕过 API）测试模型，逐层缩小。
3. **对比差异**：对比"正常"和"异常"两个场景的代码路径、参数配置、环境变量。例如 `_run_inference` 和 `_run_inference_clone` 的 `cwd` 差异。
4. **检查文件内容**：不要依赖 pydub 等"智能"工具，直接用 `xxd`/`hexdump` 或 Python 读文件头字节。
5. **替换测试工具**：当 pydub 报告"有声音"但用户听到无声时，用 `wave` 模块（严格 WAV 格式）验证头部。
6. **独立验证**：从 CI/测试环境中剥离出最小复现用例（如直接运行 `infer_onnx.py` 加上特定参数）。

### 3.3 常见误区

| 误区 | 后果 | 纠正 |
|------|------|------|
| "pydub 说音频正常" | 掩盖了 `.wav` 文件内容为 MP3 的 bug | 文件头部字节校验 |
| "后端 API 返回 200" | 忽略了子进程退出的无声降级 | 检查返回码和 stderr |
| "女声正常所以男声也正常" | 男声最后一句缺失未被早期发现 | 所有音色独立测试 |
| "短文本正常" | 长文本超时和截断未被覆盖 | 边界值测试（长文本、极端参数） |

## 五、架构决策记录

| 决定 | 选项 | 选择理由 |
|--------|---------|----------|
| 子进程 vs 进程内推理 | 子进程 | 隔离 ONNX 内存（4.5GB），避免与 FastAPI 争用 |
| 子进程 timeout 策略 | 600s | 实测长文本需要 340s + 余量 |
| chunk 大小 | 40 tokens | 平衡 EOS 风险和 chunk 间停顿数量 |
| 格式转码位置 | `_run_synthesis` | 统一处理，避免中间文件格式污染 |
| 克隆成功后自动切 TTS | 立即切换 | 用户可直接开始使用，不需等待预览 |
| 并发策略 | `max_workers=1` | 防止多子进程内存耗尽导致系统死机 |
| CPU 线程数 | `--cpu-threads 2` | 实测 4 线程对推理速度提升有限，但内存/线程开销倍增 |

## 六、改进方向

1. **GPU 加速**：ONNX Runtime 支持 CUDA/ROCm，可将推理速度提升 10-20 倍
2. **守护进程模式**：将子进程改为常驻后台的推理服务进程，通过 stdin/stdout 或 gRPC 通信。避免每次请求都重新加载模型（节省 ~30s 加载时间 + 4.5GB 内存）
3. **共享内存 / session 复用**：如果保留子进程方案，可研究 ONNX Runtime 的 session 共享机制，在多进程间复用模型权重，减少内存占用
4. **流式合成**：实现在生成过程中逐步返回音频，减少用户等待感知
5. **替代子进程**：`infer_onnx.py` 的进程内调用已在 `onnx_tts_runtime.py` 中实现，但因兼容性（torchaudio 后端）和稳定性（进程隔离）选择了子进程。后续可修复进程内路径，消除子进程带来的双倍内存开销
6. **长文本分段**：前端可按段落或句子分段请求合成，避免单次超长等待
7. **错误信息透传**：将子进程的 stderr 捕获后透传到前端，方便用户理解失败原因
8. **压力测试**：建立并发负载测试，确保系统在连续合成时不会出现内存泄漏或性能退化
